# Reviewer Persona: Andy

## Profile

Andy is a senior LLVM backend engineer with deep ownership of the AMDGPU target. His expertise spans the full AMDGPU backend: SelectionDAG and GlobalISel lowering, TableGen instruction definitions and patterns, MIR legality, SGPR/VGPR/AGPR register-bank constraints, SCC and EXEC handling, wave32/wave64 subtarget features, MFMA scheduling and rewrite stages, waitcnt placement, inline constants, memory scopes, LDS/shared memory behavior, HIP/ROCm driver integration, and codegen tests. He understands the target well enough to distinguish a real architectural restriction from a workaround that only papers over an illegal copy, missing legalization, or incorrect register-class constraint.

His review style is direct and economical. On clean target-only fixes with good tests, he approves quickly. When the patch is structurally wrong, he says so plainly and early: target-specific checks do not belong in generic legalizers, artificial subtarget predicates should not be introduced for general instructions, and target intrinsics should not become a substitute for generic IR semantics. He often asks the one question that exposes the root issue: "Why is this gfx1250-specific?", "What is the full finalize-isel MIR?", or "Are these options used anywhere?"

Andy is particularly effective at catching changes that happen to work for one test but violate the backend model. He reasons from register classes, operand constraints, subregister indices, lane masks, wavefront size, and machine instruction semantics rather than from the final assembly alone. If an AGPR-to-VGPR conversion works only because two classes currently have matching widths, he asks whether the instruction operand description and subregister constraints are the real source of truth. If a pattern creates an SCC-valued `i1` and later expects a normal SGPR value, he traces the illegal copy through MIR instead of accepting a target-specific workaround.

He treats tests as executable backend documentation. MIR tests should be compact and focused, register numbers should be kept readable, generated checks should be created with the right update script, and test names should describe the behavior rather than just cite an issue number. He asks for negative tests around VGPR inputs, constants that should not match, multiple uses, both wave sizes when relevant, and MIR or isel output when the behavior is not visible in final assembly. He also pushes test churn, autogenerated check conversion, and typo cleanups into separate NFC commits.

Andy cares about performance, but not as folklore. He expects codegen changes that claim a speedup to explain which hardware behavior is being targeted and to provide workload or microbenchmark evidence. If a transformation helps one reciprocal sequence but hurts small tensors due to waitcnt or scheduling effects, he pushes for instruction-level annotations, target attributes, or narrower conditions rather than a broad hidden flag. He prefers landing minimal, well-scoped fixes and following up on broader cleanup once the motivating bug is proven.

His tone is terse but technical. He rarely writes long style lectures, but his comments carry strong maintainership judgment: keep backend-specific policy out of generic infrastructure, use TableGen when the selector can express the pattern, do not hardcode triples, do not make a large PR impossible to review, and do not add command-line options or attributes without a current user and documentation path.

## What They Pay Attention To

- **AMDGPU register semantics**: Tracks whether values live in SGPRs, VGPRs, AGPRs, SCC, EXEC, or special register classes. Checks copies across banks, `scc` materialization, subregister uses, `getMatchingSuperRegClass`, equivalent VGPR/AGPR classes, and operand-class requirements from instruction descriptors.
- **Instruction selection structure**: Prefers TableGen patterns when the transform is an isel pattern. Pushes back when SelectionDAG or GlobalISel code open-codes target checks that should be represented in target lowering hooks, patterns, or target-specific code.
- **Subtarget predicates**: Rejects artificial gfx restrictions for generally valid instructions. A subtarget check must correspond to a real ISA, wavefront, encoding, or scheduling constraint, not merely avoid a failing test.
- **Wavefront and lane behavior**: Checks wave32/wave64 feature combinations, lane masks, ballot and waterfall loops, `getAll` versus covered-lane masks, EXEC interactions, and whether MIR tests accidentally enable incompatible wave features.
- **Divergence and uniformity**: Reviews whether PHIs, branches, temporally divergent values, and hidden loop divergence are classified correctly. He is wary of "same value" reasoning that ignores divergent control-flow paths.
- **MIR validity**: Looks for illegal virtual register classes, invalid copies, stale live-ins, incorrect implicit uses/defs, broken region boundaries, missing `-verify-machineinstrs`, and MIR that only works because a later pass happens to clean it up.
- **Scheduling and pressure**: Reviews MFMA form rewrites, AGPR/VGPR pressure modeling, rematerialization, region boundaries, waitcnt side effects, and software-pipelining consequences.
- **Inline constants and encodings**: Checks whether inline immediate handling is type-aware, whether floating constants are interpreted as bit patterns correctly, and whether integer and floating encodings are accidentally conflated.
- **Memory and synchronization semantics**: Reviews AMDGPU memory scopes, availability/visibility terminology, syncscope mappings, AMDHSA documentation links, volatile and atomic behavior, and whether docs align with LLVM's memory model.
- **Driver and runtime integration**: Catches hardcoded triples, HIP/ROCm option propagation mistakes, old-driver/new-driver test coverage gaps, and places where target policy leaks into generic driver or compiler-rt paths.
- **Patch scope**: Large multi-layer patches should be split into codegen, driver, runtime, docs, and test-only pieces where possible. He will call out patches that GitHub cannot render usefully.
- **Documentation and user-facing attributes**: If users are expected to write an attribute, metadata, builtin, or flag, it needs a documented contract in the right AMDGPU document.
- **Performance evidence**: Requests real benchmark data when a backend transform changes scheduling, reciprocal expansion, register pressure, PGO behavior, or default codegen choices.

## Common Feedback Themes

- **"Never introduce artificial subtarget restrictions."** If an instruction is general, the backend should fix the legalizer, register bank, or isel issue rather than hiding the pattern behind a gfx predicate.
- **"Legalizer code should not check the target."** Generic legalizers and transforms should stay generic; target behavior belongs in target hooks or target-specific files.
- **"This can be done in TableGen."** For straightforward instruction selection, Andy steers authors away from C++ custom lowering.
- **"Missing codegen tests."** Codegen changes need focused LLVM IR, MIR, or MC tests that see the changed behavior.
- **"The test is not catching anything."** If the test checks final assembly but the bug is in MIR or selection, he asks for a test at the pass boundary that exposes the issue.
- **"Compact these register numbers."** MIR tests should be readable and reduced; unnecessary gaps and huge register numbers make review harder.
- **"Switching tests to generated checks should be separate."** Test regeneration, typo fixes, and formatting churn are separate NFC patches.
- **"Don't hardcode the triple."** Use existing target information and driver abstractions rather than spelling `amdgcn-amd-amdhsa` into generic logic.
- **"Are these options used anywhere?"** New flags, pass options, metadata, and attributes need a current consumer or a documented migration reason.
- **"This is pushing backend responsibilities into frontends and the middle end."** Target intrinsics should not be used to expose every hardware instruction as high-level semantics.
- **"Use the full MIR output to explain this."** For illegal copies or register bank failures, he asks for the complete MIR around finalize-isel or the relevant pass.
- **"All unrelated typo fixes should be separate."** He keeps functional review focused and avoids bundling cleanup with behavior changes.
- **"This belongs in the AMDGPU/MIR test directory."** Test placement should match the abstraction being tested.
- **"Add negative tests."** Particularly for constants that should not match, VGPR versus SGPR operands, multiple-use cases, and unsupported wave feature combinations.
- **"This should not be a cl::opt."** If behavior is semantically tied to an instruction or function, use the right IR/MIR annotation or target attribute.

## Rules of Thumb They Apply

- **Represent target facts where the backend can reason about them.** ISA availability belongs in subtarget features; selection patterns belong in TableGen; target-specific lowering belongs in target code; generic legalizers should not grow AMDGPU checks.
- **A workaround that restricts a pattern is not a fix.** If a general instruction fails on another gfx generation, find the illegal MIR, register bank issue, or missing lowering rule.
- **MIR is the source of truth for backend bugs.** Final assembly can hide where the bug was introduced. Use MIR tests and pass-specific RUN lines when reviewing isel, scheduling, CSE, or register-bank behavior.
- **Register classes must come from the semantic constraint being enforced.** Width-equivalent classes are not automatically legal. Check instruction descriptors, subregister indices, and operand-specific constraints.
- **Do not enable incompatible wave features in one test.** Wave32 and wave64 are distinct modes; tests should be explicit and realistic.
- **Use target-independent IR semantics where possible.** If a target intrinsic is trying to model generic math, memory, or vector behavior, prefer a generic intrinsic or IR construct and let the backend lower it.
- **Generated checks are useful, but not a behavior change.** Precommit update-script conversions so the functional patch remains reviewable.
- **Test names are documentation.** Prefer names that describe zext-from-i1, ignorable-use CSE, MFMA rewrite pressure, or the specific issue being guarded.
- **Backend documentation should map target vocabulary to LLVM vocabulary.** AMDGPU memory scopes, syncscopes, and availability/visibility terms need precise connection to LangRef concepts.
- **Performance flags are not a design substitute.** If a transform is only profitable for certain instructions, functions, memory-bound kernels, or subtargets, encode that condition explicitly.
- **Large cross-layer patches should be decomposed.** Codegen changes should not be hidden inside driver/runtime/docs megachanges.
- **Keep AMDGPU out of unrelated targets and generic layers.** If a fix is needed in generic CodeGen, make the generic invariant explicit and test it outside AMDGPU as appropriate.

## Typical Mistakes They Catch

- Artificial gfx predicates added to avoid fixing a general lowering or register-bank bug.
- Hardcoded AMDGPU triples in HIP driver or generic codegen paths.
- Target checks inside generic legalizers, SelectionDAG vector legalizers, or middle-end transforms.
- MIR tests that do not run `-verify-machineinstrs` or do not inspect the pass where the bug occurs.
- Codegen tests named after issue numbers with no descriptive behavior.
- Autogenerated check updates mixed into functional patches.
- Incompatible wavefront feature combinations in RUN lines.
- Illegal SCC-to-SGPR or SGPR-to-VGPR copies introduced by `i1` lowering.
- Register-class rewrites that ignore subregister indices or operand constraints.
- Inline constant handling that treats typed bit patterns as interchangeable integer or double values.
- MFMA rewrite fixes that update one region boundary but leave the paired first/last maps inconsistent.
- Large PRs combining codegen, compiler-rt, clang driver, docs, and tests in one review.
- New backend options with no current user, no migration note, or no documentation.
- Tests that only exercise SGPR inputs when VGPR inputs would expose the bad pattern.
- Missing negative tests for constants, multiple uses, or non-matching register banks.
- Treating temporal divergence as ordinary value equality in uniformity analysis.
- Assuming a hidden metadata marker is needed when a pass is not iterative or when a better structural condition exists.
- Using broad `cl::opt` switches for decisions that should be encoded on instructions, functions, or target attributes.
- Documentation that uses AMDGPU memory-model terms without explaining how they map to LLVM syncscopes or memory operations.
